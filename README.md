# L'Olympus — Virtual Software House (v2)

An autonomous, gamified software house. Thirteen AI roles pull tasks from a shared PostgreSQL
task pool, build real artifacts on disk, and report up in real time through a Phaser office,
Monaco code feed, Kanban board, and Human-in-the-Loop chat.

## Stack

- **Web**: React Router v7 (framework mode) + Vite + Tailwind v3
- **DB**: Drizzle ORM against cloud Postgres (DigitalOcean)
- **Agents**: `@mastra/core` + `@mastra/memory` + `@mastra/pg`
- **Models**: OpenRouter (4 tiers — fast / reasoning / coding / vision)
- **Realtime**: in-process `EventEmitter` bridged to browsers via SSE,
  persisted to a `olympus_events` table and mirrored across processes with
  `LISTEN/NOTIFY`
- **QA**: Playwright (headful Chromium)
- **Runtime**: Node `child_process` for generated projects

## Prerequisites

- Node 20+
- pnpm 9+
- A reachable Postgres (the repo targets the one encoded in `.env` → `DATABASE_URL`)
- An OpenRouter API key

## Setup

```bash
pnpm install
# fill in DATABASE_URL + OPENROUTER_API_KEY in .env
pnpm dev            # migrations run automatically on first startup
```

If Postgres uses a self-signed cert (e.g. DigitalOcean managed DBs), the app already relaxes
TLS verification in `server/db/client.ts` and `drizzle.config.ts`.

### Database Migrations

This project uses **Drizzle ORM** migrations. When you need to change the database schema:

1. **Edit** `server/db/schema.ts` (the TypeScript source of truth)
2. **Generate** the migration: `pnpm db:generate`
3. **Review** the generated `.sql` file in `server/db/migrations/`
4. **Commit** the migration files with your code
5. **Migrations run automatically** when the dev server starts

⚠️ **Never**:
- Edit `.sql` migration files by hand
- Run `pnpm db:push` (use migrations instead)
- Skip running `pnpm db:generate` after schema changes

See [`MIGRATIONS.md`](./MIGRATIONS.md) for the complete workflow and troubleshooting.

## Run

```bash
pnpm dev
```

Open <http://localhost:3100>. You'll land on `/projects`:

1. Fill in a **name** and **brief**, click *Create project* → redirect to `/projects/<id>`.
2. The control room shows an empty office + empty Kanban. The 13 role daemons
   started automatically with the server and are polling the `olympus_tasks` table.
3. Seed a ticket from the CLI:
   ```bash
   pnpm seed:smoke
   ```
   Or `POST /api/tasks` from any HTTP client.
4. Watch the Kanban card move `todo → in-progress → done`, the sprite walk to
   the task board in the Phaser *office* tab, code chunks stream into the
   *editor* tab, and a chat entry appear on the right as each role finishes.

## Human-in-the-Loop

When an agent calls `request_human_input`, the task flips to
`blocked-needs-input` and a yellow `?` bubble appears above the sprite. Type
a reply in the chat sidebar — the API route (`/api/chat`) writes to the Mastra
memory thread and unblocks the task back to `todo`. The next poll cycle
re-claims it and resumes.

## Follow Mode

Click any role chip in the chat sidebar to subscribe the *editor* + *office*
panes to SSE events from that role only. Click again to unfollow.

## API Endpoints

### Task Management

**List all tasks in a project:**
```
GET /api/tasks?projectId=<project-uuid>
```
Returns an array of tasks for the specified project.

**Get a single task by ID:**
```
GET /api/tasks?id=<task-uuid>
```
Returns a single task object. Returns `404` if the task does not exist.

**Create a new task:**
```
POST /api/tasks
Content-Type: application/json

{
  "projectId": "<uuid>",
  "role": "pm",
  "title": "Task title",
  "description": "Optional description",
  "dependsOn": ["<uuid>"]  // optional array of task IDs this depends on
}
```

**Update a task:**
```
PATCH /api/tasks
Content-Type: application/json

{
  "id": "<task-uuid>",
  "status": "todo|in-progress|pending-review|blocked-needs-input|done|failed|skipped",
  "title": "Updated title",  // optional
  "description": "Updated description"  // optional
}
```

## Layout

```
  components/        Control Room UI (kanban, editor, office, hitl, terminal)
  lib/               event-bus, follow-mode, SSE hook
  styles/
server/
  db/                Drizzle schema + queries + migrations
  const/             role registry, colors, labels
  mastra/            runtime (Memory + PostgresStore) + role prompts + agent factory
  tools/             FS / StreamCode / HITL / DB / Runtime / Playwright
  daemon/            claim / loop / dispatcher
  workspace/         per-project file paths

```

## Scripts

| script | what |
| --- | --- |
| `pnpm dev` | Vite dev server on `OLYMPUS_WEB_PORT` (default 3100), boots the 13 role loops |
| `pnpm build` / `pnpm start` | production build + Node server |
| `pnpm db:generate` | regenerate Drizzle migrations from `server/db/schema.ts` (run after schema changes) |
| `pnpm db:verify` | verify database schema matches the TypeScript definition |
| `pnpm db:push` | ⚠️ push schema directly (not recommended; use migrations instead) |

| `pnpm typecheck` | `react-router typegen` + `tsc --noEmit` |

## Env vars (see `.env`)

- `DATABASE_URL` — cloud Postgres
- `OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL`, `OPENROUTER_APP_TITLE`, `OPENROUTER_HTTP_REFERER`
- `MODEL_TIER_FAST` — OpenRouter model id for fast, lightweight operations (PM kickoff, quick decisions)
- `MODEL_TIER_REASONING` — OpenRouter model id for complex reasoning (architecture design, planning)
- `MODEL_TIER_CODING` — OpenRouter model id for code generation (backend/frontend implementation)
- `MODEL_TIER_VISION` — OpenRouter model id for vision/analysis tasks (QA, testing)
- `MODEL_TIER_COMPLEX` — OpenRouter model id for big-picture triage and HITL filtering (CTO role only)
- `MODEL_TIER_PLANNING` — OpenRouter model id for detailed planning (PM, Architect, Tech Lead roles)
- `OLYMPUS_WEB_PORT` — default 3100
- `OLYMPUS_WORKSPACES_DIR` — default `./workspaces`
- `OLYMPUS_EMPLOYEE_POLL_MS` — default 5000
- `OLYMPUS_EMPLOYEE_POLL_MS_<ROLE>` — per-role override (e.g. `OLYMPUS_EMPLOYEE_POLL_MS_BACKEND_DEV`)
- `OLYMPUS_DISABLE_WORKFORCE=true` — boot the web without the 13 daemons (useful for UI hacking)
- `PLAYWRIGHT_HEADLESS=true` — run QA headless

## Model Tier Configuration & API Quotas

⚠️ **CRITICAL**: The default free tier models in `.env` have **extremely limited quotas** and will be exhausted within 1-2 projects. For any testing beyond a single smoke test, you **must upgrade to paid tier models**.

### Free Tier Limitations
- `google/gemini-2.5-pro:free` — ~15,000 tokens/day
- `openai/gpt-oss-120b:free` — Limited daily requests

A single project (PM + Architecture + Planning + Frontend implementation) typically consumes 50k-100k tokens, exhausting free quotas immediately.

### Recommended Production Models
```env
MODEL_TIER_FAST=openai/gpt-4o-mini
MODEL_TIER_REASONING=openai/gpt-4-turbo
MODEL_TIER_CODING=openai/gpt-4
MODEL_TIER_VISION=openai/gpt-4-vision
MODEL_TIER_COMPLEX=openai/gpt-4-turbo
MODEL_TIER_PLANNING=openai/gpt-4-turbo
```

### API Quota Management
- Monitor your OpenRouter account for quota usage
- Expect ~50k tokens per small project (to-do list app)
- Expect ~200k+ tokens per large project (full-stack application)
- Set up alerts if using a shared API key across multiple installations
- Consider implementing per-project token budgets in future versions

## Notes & gotchas

- `DATABASE_URL` in the committed `.env` is a placeholder (`show-password`). Fill in the real
  password before `pnpm db:push` or the workforce loops will log auth failures.
- **Model tier environment variables are required** — all 6 `MODEL_TIER_*` variables must be set or the PM/Architect/TechLead loops will fail on startup.
- The workforce boots as a side-effect of `app/entry.server.tsx`, guarded by
  `globalThis.__olympusBootstrap` so Vite HMR doesn't spawn duplicates.
- Code streaming is chunk-paced at 15ms per 256 chars so humans can actually watch;
  final files are still written atomically after the stream completes.
- `@mastra/pg` manages its own `mastra_*` tables for threads/messages; our
  Drizzle schema uses an `olympus_*` prefix and will not collide.
- When modifying `.env` during development, restart the server completely — Vite HMR can kill agent processes mid-execution, leaving tasks in `in-progress` state.
